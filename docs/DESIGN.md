# デザイン設計ドキュメント

このドキュメントは keishis.github.io の設計判断の根拠をまとめたものです。
実装の変更・コンポーネント追加・レイアウト変更を行う際は、ここに記載された原則との整合性を確認してください。

---

## 1. ペルソナ

### 優先度 A：学術関係者

**代表的な訪問シナリオ**
- 学会・研究会でのポスター発表後、名刺の代わりに URL を渡された
- 就活・進学活動において、学外の研究者のバックグラウンドを確認している

**目標**
1. 研究テーマと専門分野を短時間で把握する
2. 連絡先（メール）にたどり着く
3. 主要な業績・発表の詳細（abstract・外部リンク）を確認する

**言語**：基本的に日本語話者。国際学会を意識して英語でも同等の情報を提供する。

**成功条件**：研究テーマ・直近の主要業績・連絡先が 1 分以内に把握できる。

---

### 優先度 B：産業界採用担当

**代表的な訪問シナリオ**
- LinkedIn / GitHub からのプロフィール確認
- 就活エントリー後に採用担当が候補者情報として参照する

**想定職種**：ML エンジニア・企業研究所のリサーチャー・ソフトウェアエンジニア（Rust/Julia）

**目標**
1. 技術的な幅（Julia, Rust, Python など）を把握する
2. ポートフォリオ・実装物の質を評価する

**成功条件**：「面白い技術をやっている人だ」と感じ、ポートフォリオや GitHub リンクにたどり着ける。

---

### 優先度 C：技術ブログ読者

**代表的な訪問シナリオ**
- 検索や SNS 経由で特定の記事に流入する
- 同分野の研究者・エンジニアがアウトプットを継続的に追う

**想定読者**：統計・機械学習分野の研究者・学生、Julia / Rust / Python 技術に関心があるエンジニア

**ブログの位置づけ**
- 技術的実装・研究成果の詳細なアウトプット
- 研究・エンジニアリング活動の周辺情報の記録

**成功条件**：記事のコンテンツを快適に読み終え、関連記事やプロフィールへ自然に誘導される。

## 2. デザイン原則

### 原則 1：シンプルかつモダン
装飾よりも情報の明瞭さを優先する。コンポーネントは最小限の視覚的ノイズで情報を伝える。派手なアニメーション・過剰なグラデーション・コーポレートサイト的な重厚さは避ける。

### 原則 2：研究者としての誠実さ
「面白いことをやっている」という印象を、誇張ではなく実績と技術の質で伝える。コンテンツが主役であり、デザインはそれを引き立てる脇役。

### 原則 3：情報階層の明確さ
ペルソナ A が最優先。トップページは「誰が・何を研究しているか・どう連絡するか」が最初の画面で完結する。詳細情報（論文一覧など）は独立したページに分離する。

### 原則 4：一貫したトークン使用
スペーシング・タイポグラフィ・カラーはすべてデザイントークン経由で指定する。コンポーネント内に直値（`4rem`, `#22c55e` など）を書かない。

## 3. 情報アーキテクチャ

### トップページの情報優先度（above the fold）

| 優先度 | 情報 | 理由 |
|--------|------|------|
| 1 | 氏名・連絡先（メール） | ペルソナ A・B の最終目標はここにたどり着くこと |
| 2 | 研究テーマ・専門分野 | 「誰か」の次に「何をしているか」が来る |
| 3 | 主要論文・発表（抜粋） | 学術的信頼性の根拠。詳細は Publications ページへ誘導 |
| 4 | ポートフォリオ（抜粋） | ペルソナ B への訴求。技術的な幅を示す |

### ページ構成

```
/                    トップページ（ヒーロー・主要業績・ポートフォリオ抜粋・経歴・Changelog）
/publications        論文・発表一覧（フル）
/portfolio           ポートフォリオ + 資格一覧
/[lang]/blog/        ブログ記事一覧
/[lang]/blog/[slug]  ブログ記事
```

**設計方針（現状からの差分）**：
- Publications の全件一覧はトップページから切り離し `/publications` へ独立。トップには主要論文・発表のみ抜粋表示し、一覧ページへ誘導。
- `/publications` は言語切り替えを行わない単一ページ。国際会議・ジャーナルの情報は英語のみ、国内ワークショップの情報は日本語のみで表示する（掲載言語はコンテンツの性質に従う）。
- Certifications はジャンルは異なるが `/portfolio` ページに統合。トップには表示しない。
- Changelog はトップページに残すが、**折りたたみ（アコーディオン）形式**で表示。デフォルトは閉じた状態とし、見たい人だけが展開できる。ナビゲーションリンクは設けない。

### ナビゲーション

```
Home | Publications | Blog | Portfolio
```

連絡先はヒーローセクション内に常時表示。フッターに SNS リンクを維持。

## 4. コンポーネント設計方針

### 責務の分離

```
src/components/
├── primitives/     状態・データなし。btn, card, chip, badge など
├── patterns/       データを props で受け取り表示する。PubCard, PostCard など
├── sections/       データをパターンに配分する。HeroSection, PublicationsSection など
└── layouts/        ページ全体の骨格。BaseLayout など
```

- `primitives/` は外部データに依存しない
- `patterns/` はデータ取得を行わない（props のみ）
- `sections/` が唯一データロジックに触れる層

### Props の設計
- `sections/` のコンポーネントは `lang` を受け取り、内部で i18n を解決する
- `patterns/` のコンポーネントは言語非依存な型付きデータを受け取る

## 5. ビジュアル・トーン指針

### カラー
- アクセントカラー（グリーン系）は強調・インタラクション要素のみに使用
- 背景・テキスト・ボーダーはすべてセマンティックトークン（`--color-bg`, `--color-fg` 等）を使用

### タイポグラフィ
- 見出しは `--font-display`（Space Grotesk）
- 本文は `--font-sans`（Inter + Noto Sans JP）
- ブログ記事・論文 abstract は `--font-serif`（Merriweather）も選択肢

### 余白・密度
- セクション間の余白は `--spacing-section` トークンで統一
- 情報密度は「読み飛ばせるが、読めば詳細がある」構造。ペルソナ A は 30 秒で把握、ペルソナ C は 10 分読む、両方に対応できる密度設計。

### 避けるべきパターン
- 背景に強いグラデーション・大きな装飾的イラスト
- 過剰なアニメーション（hover 程度は可）
- コーポレートサイト的な「実績数字を大きく表示」するレイアウト
